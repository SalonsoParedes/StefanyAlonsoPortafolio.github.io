Blended Learning Program · Mango × Concentrix
Building BackOffice Competency from Zero
A blended onboarding program for a legacy warehouse system used by Mango's global customer service team
Articulate Storyline 360 Blended Learning Legacy Systems Stakeholder Management Multi-site End-to-end ownership

Mango's customer service team at Concentrix managed post-sale incidents for one of Europe's largest fashion retailers. New agents had to operate across multiple systems simultaneously — including wIntegrate (WTG), a legacy terminal system with no graphical interface, navigated entirely by keyboard shortcuts. Learning it wrong didn't just affect the agent: it affected real Mango customers.

The challenge
Making a 1990s terminal learnable in days, not weeks

wIntegrate has no GUI, no mouse, no icons. Every action requires knowing the right key sequence at the right moment. Agents also had to coordinate between WTG, MANTO, WCM, and Salesforce — four systems with their own logic, in the right order, or real orders would be broken.

My role
Full program ownership — from design to production floor

I designed the e-learning curriculum in Storyline, coordinated directly with Mango to request individual system credentials for each agent, facilitated the supervised practice sessions, and developed all supporting reference materials. No external brief — I mapped the content myself from operational knowledge.

Program structure

The program was deliberately blended — e-learning alone couldn't prepare agents for a live production environment. Each agent needed to handle real cases before going solo.

Phase 1 — E-learning
Module #7 of a structured onboarding curriculum, covering 8 BackOffice procedures: address modification, WTG and WCM cancellations, bank detail entry, payment extension, return processing, and blocked return cancellation. Each section used step-by-step annotated screenshots of the real system, bilingual EN/ES reference tables, conditional logic explanations, and interactive knowledge checks.
Phase 2 — Credential setup
I coordinated directly with Mango to request individual production system access for every agent in training — a cross-functional stakeholder process that required tracking, follow-up, and timing coordination with the trainer schedule.
Phase 3 — Supervised practice
Each agent worked through real pending cases (actual cancellations, address changes, stuck returns) under trainer supervision. Not simulated — real tickets, real systems, real consequences. Agents had to demonstrate competency before going live.
Phase 4 — Consolidation activity
Using a curated list of real order numbers, agents completed a structured Excel exercise logging every error type found in General Return Query — building both technical fluency and diagnostic thinking. The completed document became a personal reference tool they kept on the floor.
Key design decisions
01
Annotated real-system screenshots, not simulations
Rather than recreating the system in a simulator, I captured the actual terminal screens and annotated them with arrows, colour-coded borders, and callout boxes. Agents learned to read a real screen, not a cleaned-up version of it.
02
Visual keyboard interactions for a keyboard-only system
Since wIntegrate is 100% keyboard-driven, I designed assessment interactions using images of physical keyboards as clickable elements — testing key knowledge in a format that matched the actual cognitive task.
03
Conditional logic made visible
Critical procedures had real consequences if done in the wrong order. I designed parallel scenarios showing both paths: "If you block in WTG but skip MANTO, X happens. If you cancel in MANTO but skip WTG, Y happens." Not just rules — logic.
04
Bilingual reference tables for international teams
WTG displays status codes in English. The team operated in Spanish. I built bilingual status tables (EN/ES, short and long descriptions) embedded in the module — usable during training and as post-training reference on the floor.
Why this matters beyond the module
This wasn't instructional design as content production. It was instructional design as operational readiness.

Every agent who completed this program had already resolved real customer cases before going live. The blended structure, the production credentials, the error-logging exercise — these weren't extras. They were the difference between an agent who could follow a procedure and one who could handle anything that came in.

Multilingual E-learning · Mango × Concentrix
The Protocol That Had to Work in Five Languages
An interactive e-learning module covering Mango's customer communication protocol across five markets and five languages
Articulate Storyline 360 Multilingual Decision Trees Audio Design 5 Markets Published & deployed

Mango's customer service handled contacts across Spain, France, the UK, Germany, and Italy — each market with its own language, communication conventions, and procedural rules. A single agent might handle contacts in multiple languages. The protocol training had to be accurate, market-specific, and usable as an ongoing reference — not just a one-time course.

The scope
Five languages. One coherent experience.

ES · FR · EN · DE · IT — each with different tone conventions (tuteo vs. usted, Mr/Ms vs. first name), different escalation rules, and different written communication standards. The module used interactive tab navigation to let agents explore each market's specifics without losing the overall structure.

The complexity
Decision trees for non-obvious situations

Call verification, cross-order handling, email mismatch scenarios — these aren't linear procedures. I designed interactive branching decision trees that agents could click through to find the right action for their specific situation: who is calling, what data matches, what the options are.

Design highlights
Audio integration
Hearing the protocol, not just reading it

Call greeting scripts were paired with audio playback — agents could hear the exact tone and phrasing expected, not just read a transcript. This was especially critical for markets like Germany and France where formality conventions differ significantly from Spanish.

Interactive branching
Decisions, not procedures

The identity verification flow and email mismatch scenarios were built as clickable decision trees — agents clicked through nodes to find their situation and the correct action. This mirrors real call handling: you don't follow a script, you navigate a situation.

Reference design
Built to be consulted, not just completed

The module architecture allowed agents to jump to any section after completion. It wasn't designed as a linear course — it was designed as a searchable reference they'd return to when facing an edge case on the floor.

Scale context
Five markets. Five languages. One consistent quality standard.

The Protocol module was part of a larger onboarding curriculum covering Mango Likes You loyalty programme mechanics (including market-specific currency conversion tables), InContact telephony setup, and BackOffice operations. All modules shared a consistent visual identity and navigational logic — so agents onboarded to one could use others without relearning the interface.

03
Tool Enablement · NICE InContact / CXone
From Day One to First Call — Telephony Onboarding
A step-by-step e-learning module preparing agents to use NICE InContact from login to live call handling
Articulate Storyline 360 Tool Training NICE CXone Contact Centre Part of onboarding curriculum

When Mango's contact centre migrated from Avayas to NICE InContact / CXone, every agent needed to be operational on the new platform before handling a single live call. The challenge wasn't just showing where buttons were — it was ensuring agents understood the logic of the system: states, aux codes, agent leg behaviour, and what goes wrong if you get it wrong.

What I designed
Six-section module, zero assumptions

The module covered the full agent journey: connecting via Citrix, navigating the MAX interface, understanding status states and aux codes, handling inbound calls (accept, hold, mute, transfer), making outbound calls with country prefix logic, and understanding the five tabs of the interface. Each section used numbered lateral navigation so agents could move at their own pace and return to any step.

The design principle
Show the system, explain the logic

Screenshots of the real CXone interface were annotated with contextual callout boxes — not just "click here" but "this is why." For example: the Agent Leg should always be inactive (not connected) — a counterintuitive setting that agents frequently got wrong. The module called this out explicitly, with a visual reminder, before they ever saw a real call.

⚠
A critical moment in the module: InContact doesn't notify agents of incoming calls with a pop-up — only with sound. The module included a prominent reminder to enable browser audio volume before going available. This single instruction prevented a category of missed-call errors that had caused quality issues during the migration period.

Program Design Exercise · CX Onboarding Architecture
Designing a Training Department from a Blank Page
A strategic onboarding plan for a CX team with no existing training infrastructure — built as a design exercise
Program Architecture KPI Design Blended Learning Onboarding Strategic design exercise

Given the brief of designing a complete CX onboarding program for a SaaS company with no existing training structure, I built a two-week hybrid program from scratch — covering not just the learning journey but the measurement framework, the support model for underperforming agents, and the infrastructure needed for ongoing development after onboarding ends.

Week 1 — Theoretical foundations (25h)
Building capability, not just familiarity

Five structured days covering company culture, product and customer journey, tools (CRM, helpdesk, knowledge base), communication and conflict handling, escalation procedures, and customer typology. Each day included a deliberate mix of input, practice activity, and knowledge check. Day 2 featured a creative exercise — "The Drawing Challenge" — where agents replicate geometric figures from verbal instructions only, building active listening skills in a memorable, low-stakes context.

Week 2 — Practical application (25h)
From observation to ownership

A four-day shadowing progression: first observing senior agents handling real cases, then moving to reverse shadowing (new agent handles, senior guides). Day 10 included structured reverse shadowing evaluated against six KPIs — written communication, procedural accuracy, tone and empathy, software navigation, problem solving, and positive attitude — followed by a modular final exam requiring 80% to pass.

The measurement framework

A training program without measurement is just content delivery. I designed a post-onboarding KPI tracking framework covering six metrics:

Support for underperformers
No agent left behind without a plan

Agents below threshold received a structured support path: one-to-one feedback with the trainer, optional shadowing extension (additional 2 days), custom reinforcement plan with assigned e-learnings and focused practice cases targeting their specific low-score areas. Not generic remediation — targeted intervention.

Post-onboarding infrastructure
Building for what comes after day one

The program included recommendations for sustainable L&D infrastructure: a trainer rotation model for diverse perspectives, a self-paced e-learning library for ongoing development, a buddy system pairing new hires with experienced agents for their first two weeks on the floor, and a personal KPI dashboard so agents could track their own progress against benchmarks.

What this exercise demonstrates
I don't wait to be given a training brief. I identify the problem, design the system, and build for what the organisation needs next.

This program was designed under time constraints as a strategic exercise, covering onboarding architecture, measurement design, remediation planning, and long-term L&D infrastructure — in a format ready to present and defend. It reflects the same thinking I bring to real programs: start with the business problem, design the learning system, measure what matters.

Customer call simulator

Hi! I'm going to walk you through a real customer call. A shopper's order has failed to arrive twice, right before an important event, and they are upset. Your job is to choose how the agent responds at each step. Watch the temperature gauge: empathy and ownership cool it down, defensiveness heats it up. Ready when you are.

Step 1 of 3 · How does the agent open?

Design note

This is the same logic I build into branching e-learning: every choice carries a consequence, the feedback is immediate and explains the why, and the environment becomes a second feedback channel. The temperature meter and the room turning hostile do the job of a results slide, but they let the learner feel the cost of a misstep instead of reading about it. Empathy first, own the failure, resolve in the moment, close with accountability. That is the de-escalation model underneath the game.
